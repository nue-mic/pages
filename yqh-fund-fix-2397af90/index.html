FIX-3记账并发「假失败」且人工重试不可自愈

💣 休眠雷 · 新缺陷
触发：开真实分账后取证：取证C性质：文档反当既定行为
问题金额层（uk_idem）安全不双花，但状态层被并发污染成「假失败」且永久钉死。三点联动：(a) 三处成功分支忽略 markSuccess 的 CAS 返回行数，仍无条件 accountDivisionSuccess → 多路径同时入账；(b) markAccountingFailed 无 accounting_status 守卫（对比 markAccountingSuccess 有），把已成功(1)覆写回失败(2)；(c) 人工 retryAccounting 因 payable 已在 available、markPayableInAvailable 返回 0 被当硬失败抛 IllegalStateException → 再标失败 → 永久钉死。
根因CAS 返回值被丢弃（放大器）+ 守卫不对称 + 把「payable 已在目标态」的幂等成功误判为硬失败。

FIX-4提现 success=false 被无条件当成功

💰 资金安全 · 新发现

根因把「已受理待回调」误当「已终态成功」。

  1. payout 返回后按 resp.isSuccess() 分流：true → payFinalize；false → 保持 PAYING 等异步回调/查单，由超时 Job 兜底。触碰 SupplierWithdrawalServiceImpl.java:262 / 274-275。

FIX-5提现监听器自调用 payFinalize，事务注解失效

A.1 · 号源方案一：Redis INCR（推荐）

新增 service/channel/no/PaymentChannelNoGenerator.java 接口（nextNo()，失败抛异常、禁止返回 null）+ RedisPaymentChannelNoGenerator 实现：

RedisPaymentChannelNoGenerator.javaJava
@Component
@ConditionalOnProperty(prefix = "yqh.balance.payment-channel-no", name = "strategy",
        havingValue = "redis", matchIfMissing = true)
public class RedisPaymentChannelNoGenerator implements PaymentChannelNoGenerator {
    private static final DateTimeFormatter FMT = DateTimeFormatter.ofPattern("yyyyMMddHHmmss");
    private static final ZoneId ZONE = ZoneId.of("Asia/Shanghai");
    // INCR + PERSIST 作为单个 Lua 脚本原子执行；PERSIST 清除遗留 TTL
    private static final DefaultRedisScript<Long> NEXT = new DefaultRedisScript<>(
        "local v = redis.call('INCR', KEYS[1]) redis.call('PERSIST', KEYS[1]) return v", Long.class);
    private final StringRedisTemplate redis;
    private final String key; // yqh:{env}:payment-channel-tx:no:global-seq

    public String nextNo() {
        Long seq = redis.execute(NEXT, List.of(key));
        if (seq == null || seq <= 0) // 号源不可用直接失败，禁止降级回 AtomicLong
            throw exception(PAYMENT_CHANNEL_TRANSACTION_CREATE_FAILED, "REDIS_NO_GENERATOR", key);
        // %019d 只补齐宽度、不取模，序号永不回绕
        return "PCT-" + LocalDateTime.now(ZONE).format(FMT) + "-"
             + String.format(Locale.ROOT, "%019d", seq);
    }
}
application-*.yamlYAML
yqh:
  balance:
    payment-channel-no:
      strategy: redis
      redis-key: yqh:${DEPLOY_ENV:local}:payment-channel-tx:no:global-seq
  • Key：全租户全 pod 共用一个序列，不得在 key 加 pod ID（否则又变多个局部号源）。
  • TTL：不设 TTL + Lua PERSIST 清残留；该 key 配不可淘汰 + AOF + 主从/故障转移。
  • 并发正确性：Redis 单 key 命令串行化，每次 INCR 得不同 seq；号里 seq 不取模 → 即使多 pod 同秒同前缀也不会撞完整号。
  • 异常：Redis 挂 → 抛业务异常由调用方回滚；DB uk_tenant_no（yqh_balance_r_alter.sql:195）+ 三次重试作最后防线。

A.2 · 号源方案二：雪花（备选）

新增 SnowflakePaymentChannelNoGenerator（strategy=snowflake）：41 位时间戳 + 5 位 datacenterId + 5 位 workerId + 12 位序列，synchronized nextId()，时钟回拨抛异常关闭，worker-id/datacenter-id 由 StatefulSet ordinal 显式注入（禁止随机/默认 0）。仅适合能可靠分配节点 ID + 监控时钟的部署。

A.3 · 推荐结论

Redis INCR。仓库已由多租户 starter 间接引入 Redis（yudao-module-yqh-balance/pom.xml:75），单原子计数器、无需维护 worker ID、规避雪花的节点 ID 重复与时钟回拨风险。生产用持久 key + AOF + 主从 + 禁淘汰，保留 DB 唯一索引与三次重试兜底。

A.4 · 统一插入模板 insertOrReuse

新增错误码 PAYMENT_CHANNEL_TRANSACTION_CREATE_FAILED = 1_040_007_007（ErrorCodeConstants.java）；selectByIdempotencyKey 改为显式带 tenant_id（@TenantIgnore + SQL 显式条件）；catch 回查走独立 PaymentChannelTransactionLookupService（@Transactional(REQUIRES_NEW)）。核心模板四步逻辑：

  1. 先查幂等键 — 命中 = 真幂等复用，直接返回 created=false。
  2. insert — 失败按 no 碰撞重试（初次 + 最多 3 次 = 最多 4 次 insert）。
  3. 新事务回查区分冲突来源 — REQUIRES_NEW 再查幂等键，命中 → 真幂等 created=false。
  4. 幂等键未命中 → 判定 no 碰撞 — 未超重试上限则重新 generateNo 重试，超限抛 CREATE_FAILED。
PaymentChannelTransactionServiceImpl#insertOrReuseJava
private InsertOutcome insertOrReuse(String scene, Long tenantId, String key,
        Supplier<PaymentChannelTransactionDO> factory) {
    // 1) 先查幂等键：命中 = 真幂等复用
    PaymentChannelTransactionDO existing = mapper.selectByIdempotencyKey(tenantId, key);
    if (existing != null) return new InsertOutcome(existing, false);
    // 2) insert，失败按 no 碰撞重试（初次 + 最多 3 次 = 最多 4 次 insert）
    for (int retry = 0; ; retry++) {
        PaymentChannelTransactionDO candidate = factory.get();
        candidate.setNo(generateNo());
        try {
            mapper.insert(candidate);
            return new InsertOutcome(candidate, true);
        } catch (DuplicateKeyException ex) {
            // 3) 新事务回查区分冲突来源（REQUIRES_NEW）
            existing = lookupService.selectByIdempotencyKey(tenantId, key);
            if (existing != null) return new InsertOutcome(existing, false); // 真幂等
            // 4) 幂等键未命中 → 判定 no 碰撞
            if (retry >= MAX_NO_COLLISION_RETRIES /*=3*/)
                throw exception(PAYMENT_CHANNEL_TRANSACTION_CREATE_FAILED, scene, key);
            // 重新 generateNo 重试
        }
    }
}
private String generateNo() { return paymentChannelNoGenerator.nextNo(); }
四入口复用 + beforeCollect 关键纠正：beforePayout / beforeDivision / beforeRefund / beforeCollect 全部复用该模板（各自 factory 填字段）。beforeCollect 当前（:166-168）对任意冲突 return false，改为返回 outcome.created() —— 真幂等重复 → false；仅 no 碰撞重试成功 → true，不再把号碰撞误判为重复收款。
